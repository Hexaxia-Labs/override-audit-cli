# Lessons from building override-audit-cli

**Date:** 2026-05-27
**Context:** Single-session build of [`override-audit-cli`](https://github.com/Hexaxia-Labs/override-audit-cli) v0.1.0 through v0.3.0. Eight detectors, 200 tests, RFC 6902 fixer with multi-op support, NDJSON change-control stream.

A few real lessons, in rough order of how surprising they were:

## The most valuable rules came from dogfood, not the spec

OA001-005 came from the design doc. OA006/007/008 came from running the v0.1.0 build against hexmetrics, seeing findings that were technically correct but missed the *actual* failure mode (the parent-coupling, the lockfile freeze, the vulnerable twin still on disk). Three of the eight rules were invented in a single dogfood session because the first five didn't explain what I was seeing.

The implication: the "ship to dogfood fast" cadence isn't an aesthetic, it's a rule-discovery mechanism.

## Severity is a UX problem, not a correctness problem

OA006 shipped at uniform `high` in v0.1.1. Hexops' postcss case (where the override was risky in principle but currently effective) made every run noisy. v0.1.2 added the platform-binary heuristic plus OA008-composite escalation. Two dogfood passes in 24 hours, both fixing false-positive *severity*, not false-positive *firing*.

If we'd gotten that wrong long-term, people would have stopped reading the output.

## The original motivating bug wasn't the most important bug in its neighborhood

OA004 was built specifically to catch HexOps' `parseInt("latest") -> NaN` regression. It's `low` severity. The bugs that turned out to matter, frozen lockfiles, fighting exact-pinned parents, vulnerable twins on disk, were adjacent but qualitatively different, and they were `high` / `critical`.

The originating frustration is rarely the worst case.

## Writing the API docs first caught a real export gap

README's "Programmatic usage" section documented `import { fix, FileLogger }`. When I did the factual audit, `src/index.ts` actually exported only `scan` and 6 types. The docs were aspirational; nothing else was exposed.

Treating the docs as a contract surfaced the gap; fixing it expanded the public API from 7 to 18 named exports, which is what an embedder actually needs.

**Doc-first as forcing function** works better than I expected.

## Schema lockdown on day one paid off through eight releases

`OverrideAuditOutput` got a snapshot test in v0.1.0. Every change since has had to be either additive (`patches?: RFC6902Patch[]` in v0.2.1) or a deliberate snapshot update.

That discipline meant the eventual `--fix`, `--log-file`, multi-op patches, and `FixReport` all landed without breaking earlier consumers, even though there *were* no earlier consumers when the test was written.

Lock the contract before there are stakeholders. The discipline is the point.

## Parser tests need fixtures from every format version found in the wild

The pnpm lockfile parser handled v6 (`/name@ver:`) perfectly and silently returned empty Set on v9 (`'@scope/pkg@ver':` quoted, bare unquoted). Caught only when scanning hexops. Adding the v9 fixture in `tests/fixtures/lockfile-pnpm-v9/` was a 10-line fix.

The actual lesson is: your test fixtures encode your assumed format, and the assumption rots.

## When you diff two query results, normalize the filter set across both ends

The post-fix rescan in `fix.ts` initially used the raw scanner output. The pre-scan was CLI-severity-filtered. An OA005.e finding that was *always* there showed up as "new" post-fix because it had been filtered out pre-fix.

Applying the same filters to the post-scan made the diff sane.

## Em dashes are a cosmetic tell

Cheap to remove, perceptually large. The doc set went from "this looks AI-generated" to "this looks like someone owns it" in one bulk substitution plus a readability pass.

Worth doing once at the end of any AI-assisted doc burst.

---

## The one to put on the wall

**The second dogfood session is more valuable than the first ten hours of spec work.**

Plan 1 was 20 tasks of prescriptive scaffolding. The session that produced OA006/007/008 was ~3 hours of "let me see what this actually finds on a real project". The latter shipped three rules and a severity-calibration story; the former shipped the substrate that made it possible. Both were necessary, but the ratio of value-to-time was wildly different.

## Hexmetrics dogfood evolution

Same project, different versions of the tool. The override block in `hexmetrics/package.json` did not change between runs:

```jsonc
"overrides": {
  "postcss": "8.5.15",
  "@esbuild-kit/core-utils": { "esbuild": "^0.25.0" },
  "@esbuild/linux-x64": "latest"
}
```

### v0.1.0 (default severity floor `low`)

```
override-audit  /home/aaron/Projects/hexmetrics  (npm)

  1 finding:

  [medium]
    OA002-FLOATING-TAG  @esbuild/linux-x64  latest  → replace
        @esbuild/linux-x64 is pinned to "latest" — every install may
        re-resolve the version, defeating the override.

exit: 1
```

One finding. Tool surfaced the floating-tag issue. No visibility into the parent-coupling problem yet.

### v0.1.1 (added OA006/OA007/OA008; default severity floor `low`)

```
override-audit  /home/aaron/Projects/hexmetrics  (npm)

  3 findings:

  [high]
    OA006-COUPLED-PLATFORM-BINARY  postcss  8.5.15  → suggest
        postcss is overridden to "8.5.15", but its installed parent
        next@16.2.6 declares it as exact (dependencies: "8.4.31"). ...
    OA006-COUPLED-PLATFORM-BINARY  @esbuild/linux-x64  latest  → suggest
        @esbuild/linux-x64 is overridden to "latest", but its installed
        parent esbuild@0.28.0 declares it as exact ...

  [medium]
    OA002-FLOATING-TAG  @esbuild/linux-x64  latest  → replace
        ...

  Skipped detectors (incomplete inputs):
    OA007  --with-registry not passed — frozen-latest check disabled

exit: 1
```

Three findings. OA006 surfaces twice (postcss + @esbuild/linux-x64). Both at uniform `high`. Same project, suddenly much louder, and in the postcss case arguably too loud (the override was working at install time).

### v0.1.1 with `--with-registry`

Added a fourth finding:

```
  [high]
    OA007-FROZEN-LATEST  @esbuild/linux-x64  latest  → suggest
        @esbuild/linux-x64 is pinned to "latest", which resolved to 0.25.12
        at install time and has been frozen there by the lockfile. The
        registry currently advertises latest=0.28.0. Your security-shaped
        pin is stale.
```

Live registry call. Confirmed the override had been silently stale by 3 minor versions.

### v0.1.2 (tiered severity)

OA006 on the platform binary stays `high` (the heuristic matches `linux` as a platform token). OA006 on postcss drops to `medium` because no platform token matches and OA008 doesn't fire:

```
  [high]
    OA006-COUPLED-PLATFORM-BINARY  @esbuild/linux-x64  latest  → suggest

  [medium]
    OA002-FLOATING-TAG  @esbuild/linux-x64  latest  → replace
    OA006-COUPLED-PLATFORM-BINARY  postcss  8.5.15  → suggest
```

Three findings, same as v0.1.1, but the severity grouping now matches actual urgency.

### v0.2.1 (multi-op patches): `--fix` actually applies

Pre-fix:

```jsonc
"overrides": {
  "postcss": "8.5.15",
  "@esbuild-kit/core-utils": { "esbuild": "^0.25.0" },
  "@esbuild/linux-x64": "latest"
}
```

Post-fix (after `override-audit --fix /tmp/hexmetrics-copy`):

```jsonc
"overrides": {
  "@esbuild-kit/core-utils": { "esbuild": "^0.25.0" },
  "next": ">=16.2.6",
  "esbuild": ">=0.28.0"
}
```

OA002 wrote `>=0.25.12` into the binary entry. OA006 then removed the binary entry and added the parent override. OA006 on postcss removed postcss and added `next: ">=16.2.6"`. OA007's later attempt to `replace` the binary entry failed (target already gone), logged as `remediation_failed`.

Three structural rewrites applied; one `failed` record for audit trail completeness; post-fix rescan reports the new state.

## Hexops dogfood evolution

Same project (`/home/aaron/Projects/hexops`, pnpm v9, `node-linker=hoisted`), different versions of the tool.

```jsonc
"pnpm": {
  "overrides": {
    "postcss": "^8.5.15",
    "ip-address": "^10.2.0",
    "vite": "^8.0.13",
    "esbuild": ">=0.28.0",
    "qs": ">=6.15.2"
  }
}
```

### v0.1.0 (before pnpm v9 lockfile parser fix)

```
override-audit  /home/aaron/Projects/hexops  (pnpm)

  No findings.
  Skipped detectors (incomplete inputs):
    OA001  lockfile missing or empty — orphan check disabled
exit: 0
```

False clean. The lockfile parser silently returned an empty Set for pnpm v9 entries; OA001 was disabled as a "graceful degradation"; nothing else had visibility into the parent-coupling.

### v0.1.0 after parser fix (pnpm v6 + v9 both handled)

```
override-audit  /home/aaron/Projects/hexops  (pnpm)

  No findings.
exit: 0
```

Genuinely clean. Five overrides, all using `>=`/`^` ranges, all pointing at packages in the resolved tree.

### v0.1.1 (with OA006 firing uniformly at high)

```
  [high]
    OA006-COUPLED-PLATFORM-BINARY  postcss  ^8.5.15  → suggest
        postcss is overridden to "^8.5.15", but its installed parent
        next@16.2.6 declares it as exact (dependencies: "8.4.31"). ...
exit: 1
```

One finding, severity `high`, even though the override was actually working at install time (only one postcss copy on disk, at the override's floor). This is what triggered the v0.1.2 severity refinement.

### v0.1.2 (after severity tiering)

```
  [medium]
    OA006-COUPLED-PLATFORM-BINARY  postcss  ^8.5.15  → suggest
exit: 1
```

Same finding, dropped from `high` to `medium`. `postcss` doesn't match the platform-binary heuristic, and OA008 doesn't fire (no vulnerable copy on disk), so the scanner-level escalation doesn't bump it back to `high`. Output is now appropriately calibrated.
