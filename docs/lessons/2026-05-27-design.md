## 1. What worked — keep doing

### 1.1 Anchor every detector to one real-world fleet example

The `hexmetrics` overrides block was load-bearing throughout the design:

```json
"overrides": {
  "postcss": "8.5.15",
  "@esbuild-kit/core-utils": { "esbuild": "^0.25.0" },
  "@esbuild/linux-x64": "latest"
}
```

Every detector decision was sanity-checked against "what does this find in hexmetrics?". Without that one concrete case, `OA005-NESTED-OVERRIDE` likely would have stayed a single vague detector instead of the five-sub-code structure (`.a-NON-NPM` through `.e-SUSPECT`) it became.

**Lesson for any sibling Hexaxia-Labs CLI:** find one project's actual problem in the fleet and design every detector against it. Synthetic examples are too easy to twist to fit your design; real ones push back.

### 1.2 Mirror parent-system contracts verbatim at the embed boundary

The CLI's change-control logging uses HexOps' `remediation_*` event vocabulary unchanged — same event names (`remediation_initiated`, `remediation_install_complete`, `remediation_install_failed`, `remediation_completed`), same field names (`findingsCovered`, `findingsResolved`, `findingsRemaining`, `previousFindingCount`, `currentFindingCount`, `installGate`, `attemptedPackages`). Source: HexOps commits `02b82a1` and `1ab55ed`.

When HexOps embeds the CLI in Plan 3, the integration is a literal pass-through: read NDJSON events from CLI's `--log-file`, re-emit via `logger.info('security', evt.event, evt.message, { meta: evt.meta })`. Zero translation step.

**Lesson:** when a tool will be embedded in a parent system, mirror the parent's contracts directly. Don't design parallel-but-similar vocabularies — that creates translation friction that compounds over time.

### 1.3 Hard scope guardrail: "purely static, no network"

Picking the static-only path at the very first brainstorming question kept v1 shippable. The vuln-aware case — "this override was added for CVE-X, now fixed in version Y" (the [pnpm#5949](https://github.com/pnpm/pnpm/issues/5949) case) — is genuinely a v2 question that requires OSV data, sync mechanics, and either depending on cve-lite-cli or duplicating its logic.

By deferring it, v1 ships a useful tool and we'll know from real dogfooding data whether the v2 work is worth doing. The decision is now data-driven, not speculative.

**Lesson:** at the brainstorming stage, identify the single biggest scope-creep risk and put a hard guardrail on it. Static-vs-vuln-aware was the lever here; for other tools it'll be a different lever, but there usually is one.

### 1.4 Sibling-pattern reuse

`src/detectors/` + Jest + hand-rolled arg parser is now established convention across three Hexaxia-Labs OSS CLIs:

- `cve-lite-cli` (OWASP Incubator) — vulnerabilities via OSV
- `supply-sentinel` (Hexaxia-Labs, going public) — supply-chain attacks
- `override-audit-cli` (Hexaxia-Labs, this project) — override hygiene

Picking the family pattern saved an architectural debate and gives HexOps a uniform `ScanSource` wrapper shape. Future tools in this family should slot in without re-deriving structure.

### 1.5 Map every decision to a memory or governance source

When the user said "keep it private for now", the [[hexaxia-group-governance]] memory immediately flagged a tension with the open-core principle ("PUBLIC = the tool that runs on the user's machine"). Surfacing that tension before locking the answer led to a cleaner resolution ("private now, public when ready") instead of "private, no plan to flip" — which would have been governance debt.

**Lesson:** scope-clarifying questions should be informed by existing governance/decision memory, not asked in a vacuum. Read the relevant memories before forming the next question.

## 2. Watch during Plan 1 implementation

### 2.1 OA005's 5 sub-codes are under one detector — let them split if they earn it

`OA005-NESTED-OVERRIDE` carries `.a-NON-NPM`, `.b-ORPHANED-OUTER`, `.c-ORPHANED-INNER`, `.d-LEAKY`, `.e-SUSPECT`. Single detector chosen because they share a parsing entry point and one finding kind is easier for users to learn.

If during coding the sub-conditions diverge enough that they're not actually sharing meaningful code — for example, `.d-LEAKY` ends up needing a substantially different lockfile traversal — splitting into separate detectors is fine. Don't refuse to split out of architectural loyalty; don't pre-emptively split for hypothetical divergence either.

### 2.2 OA004 safety heuristic is intentionally simple — flag for follow-up

Spec §4.4 and Plan Task 10 specify: same-major between installed and pin → safe `remove`; cross-major → `suggest` only. The fuller "resolved parent depends on `>=pin` natively" check is deferred per Spec §12.1.

This will likely produce some false-`suggest`s in practice (cases where removal is actually safe but we couldn't prove it). Track those during dogfooding — that data is what tells us whether the v1.x follow-up to add parent-graph traversal is worth building.

### 2.3 The `_testManifestLookup` injection in nested-override is mildly ugly

`tests/detectors/nested-override.test.ts` passes a manifest-lookup hook via a type-asserted context extension to inject test doubles. It works but it's not pretty. A cleaner approach: dependency-inject the manifest lookup into `scan()` itself and pass it down via `Context`. Fair-game cleanup during implementation if it bothers you.

## 3. For Plan 2 (Fix) and Plan 3 (HexOps embed)

### 3.1 `--no-install` is the key integration lever

HexOps already owns: install lifecycle, dev-server guards, audit verification, commit-banner UI. Plan 2 must ship `--no-install` and Plan 3 must use it. The CLI patches `package.json`; HexOps drives the rest. **One install per Apply, not two.** Plan 3 will not work if Plan 2 doesn't get this lever right.

### 3.2 Port, don't re-derive

HexOps' battle-tested helpers should be lifted directly into the CLI:

| HexOps location | Purpose | Port to |
|---|---|---|
| `src/lib/updaters/override.ts:55` `pkgJsonIndent` | Detect indent style from raw file | `src/fixer/apply.ts` |
| `src/lib/updaters/override.ts:38-58` `removeOverrideConflicts` | devDep EOVERRIDE handling | `src/fixer/apply.ts` |
| `src/lib/updaters/override.ts:165-168` install commands | npm/pnpm reconciliation strings | `src/fixer/apply.ts` |

These are not academic. They encode real failure modes from real fleet runs (the EOVERRIDE crash class documented in the HexOps `/hexops` skill's "Known Failure Modes" table). Re-deriving them risks reintroducing bugs that have already been fixed.

### 3.3 Scope boundary: NOT a typosquat detector

`override-audit-cli` will not catch malicious packages with "esbuild" in the name (e.g. `esbuild-plugin-brotli`, `esbuild-linux-arm32`). That's supply-sentinel's job. The MAL- entries observed in OSV during this session reinforce the boundary. The three tools are **complementary**:

- `cve-lite-cli` — what *known vulnerabilities* are in my deps?
- `supply-sentinel` — are any of my deps *actually malicious*?
- `override-audit-cli` — are my override declarations *coherent*?

If implementation drift tempts you toward malicious-package detection, **stop**. That capability belongs in supply-sentinel.

## 4. Process lessons (apply to future planning sessions)

### 4.1 Check sibling projects FIRST during brainstorming context-gathering

Mid-Q5, the [[hexaxia-group-governance]] memory flagged that `supply-sentinel` existed and was "publishing to npm → re-integrate into hexops as a ScanSource" (hexops #102). That was a near-miss for scope overlap — `override-audit-cli` and `supply-sentinel` could have been collapsed into one tool by a less thorough designer.

**The check should have happened at step 1** (project context exploration), not mid-Q5. The brainstorming skill's "explore project context" step needs to include "list existing repos in the same GitHub org and read their READMEs" when proposing new projects in established orgs.

### 4.2 Subagent-driven execution is the wrong tool for prescriptive plans

Plan 1 landed at 3614 lines because it shows every line of code, every test, every command. Subagent dispatch for plans this prescriptive is essentially paying for transcription. The right call — caught and switched to before execution — was a fresh `claude` session running `executing-plans` against the plan.

**Rule of thumb:** subagent ROI scales with the judgment density of each task. For tasks with multiple valid approaches, mid-implementation decisions, or significant exploration, subagents earn their keep. For tasks that are pure recipe-following, executing-plans in a fresh session is cheaper and just as effective.

Plan 2 (Fix) will have meaningfully more judgment density — atomicity edge cases, install-failure handling, conflict resolution. Reassess execution mode at Plan 2 time; don't carry forward Plan 1's answer.

### 4.3 Plan size signal: probably too detailed

3614 lines is past the inflection point where additional detail helps. Plan 2 should target ~1500-2000 lines: define interfaces concretely, show key tests, but trust the implementer to fill in obvious code between the contracts.

## 5. The esbuild near-miss (worth knowing)

Mid-session, the user surfaced OSV results for "esbuild" that included multiple `MAL-` entries (typosquats: `esbuild-plugin-brotli`, `esbuild-plugin-eta`, `esbuild-plugin-httpfile`, `esbuild-linux-arm32`, `esbuild-plugin-global-externals`). First-glance reaction was "should we rip out esbuild?"

**Data check changed the answer:**
- esbuild itself has ONE OSV advisory in its entire history ([GHSA-67mh-4wv8-2f99](https://github.com/advisories/GHSA-67mh-4wv8-2f99)), dev-server only, fixed in 0.25.0.
- The MAL- entries are *separate malicious packages* registered by third parties to ride esbuild's popularity — not compromises of esbuild itself.
- Fleet scan: clean of all five typosquats across 31 projects.
- Removing esbuild would mean swapping tsx → ts-node + vitest → jest in 10+ projects. Major work for ~zero security benefit.

Decision: keep esbuild. Memory filed at `~/.claude/projects/-home-aaron-Projects-hexops/memory/reference_esbuild_keep_decision.md` with trigger conditions for revisiting.

**Why this matters for override-audit-cli:** it's a clean illustration of the scope boundary in §3.3. `override-audit-cli` would not have flagged a typosquatted package as a malicious finding — and that's correct, because that's not its job. Conflating the two threat classes would have led to a worse tool.

## 6. TL;DR for fresh-session implementer

1. Read the spec and plan top-to-bottom before starting Task 1.
2. The plan is unusually prescriptive — most steps are transcription. Trust it.
3. The `hexmetrics` overrides block is the canonical mental anchor for "does this detector do the right thing?"
4. Don't drift into supply-sentinel's scope (typosquats) or cve-lite's scope (vulnerabilities).
5. Plan 2 (Fix) and Plan 3 (HexOps embed) will get their own sessions — don't reach for `--fix` in Plan 1.
