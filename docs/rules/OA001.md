# OA001-ORPHAN-TARGET

**Severity:** low &nbsp;·&nbsp; **Action:** `remove`

## What it catches

An entry in `overrides` or `pnpm.overrides` whose **target package is not present anywhere in the resolved dependency tree** (per the lockfile).

The override has no effect because there is no install path for it to apply to. It's accumulated debris, usually left over after a transitive dependency was dropped or renamed.

## Example

```jsonc
// package.json
{
  "dependencies": {
    "next": "^16.2.6"
  },
  "overrides": {
    "left-pad": "1.3.0"       // ← nothing depends on left-pad anymore
  }
}
```

Output:

```
[low]
  OA001-ORPHAN-TARGET  left-pad  1.3.0  → remove
      left-pad is declared in overrides but no package depends on it.
      The override has no effect.
```

## Why it matters

Orphaned overrides are mostly cosmetic noise, but they:

- **Misrepresent your security posture.** Reading the override block makes it look like `left-pad` is locked to a safe version. It isn't pinned at all; there's no install of it.
- **Decay further.** If `left-pad` later comes back as a transitive dep (e.g. someone adds a build tool that pulls it in), the stale pin will silently apply at whatever version was set years ago.
- **Add review friction.** Every reader has to mentally check whether the pin is load-bearing.

## How to fix

```bash
override-audit --fix --rule OA001    # available since v0.2.0
override-audit --fix --dry-run       # preview first
```

`--fix` emits a single-op `remove` patch and rewrites `package.json` atomically.

## When you might want to keep it

Rarely. But consider it if:

- The target is a peer of a dev tool you sometimes install (`npm install --include=optional` toggling).
- You're holding the pin in place pending a planned regression test.

If you keep it, add a comment in `package.json` explaining why so the next reader doesn't churn on it.

## Graceful degradation

OA001 needs the lockfile. Runs are silently disabled (surfaced via `skippedDetectors`) when:

- `package-lock.json` / `pnpm-lock.yaml` is missing
- The lockfile parses to an empty package set (rare; usually a corrupted file)
